# 前言

分布式算法保证分布式集群环境的一致性

# 分布式快照为什么难？

  • 系统状态是不断变化的，各个Process时间也不是完全一致的；

  • 不能让系统停下来做快照，快照对系统的负面影响应该尽量小；

  • 想象给一大群奔跑的羊拍照，由于羊群太大，一张照片没法捕捉完整，需要好几张照片合成，但是羊群不会停下来，如何能够获得一张有意义的照片？比如，每一只羊正好都只出现了一次？如果羊倌在回家前数数羊是不是少了，这种快照就是有意义的。虽然不是照片上所有的羊的位置都是同一个物理时刻的。

# 2. 分布式系统模型

2.1 分布式系统的抽象模型

  • 分布式系统由一系列Processes和通信Channel组成；
  • 认为每个Channel是有向的(一个双向连接可以认为是两个Channel);
  • 整个系统构成了一个有向图： Process是图的顶点，Channel是图的边；
  • 进程可以自己执行运算，还可以Send/Recv消息；
  • 进程可以记录状态(进程当前状态、已发送的消息，已接收的消息)。

2.2 对Channel的假设

  • 不会乱序 ，后面会看到，Snapshot Marker与其他msg之间的顺序，是不能打乱的；
  • 没有错误。

2.3 Channel的State

  • 一个Channel上已被接收的消息，是已发送消息的一个子序列(Subsequence)；
  • Channel的状态，用“已发送的消息” 减去 “已被接收的消息”表示，即： 已发 - 已收。

The state of a channel is the sequence of messages sent along the channel, excluding the messages received along the channel

  • Channel状态是由接收者来记录的。

# 3. 算法

3.1 算法是怎么来的？

3.1.1 初始思路

  • 每个Process负责记录自己的状态；
  • 每个Channel由其两端的Process分别记录(已发和已收)；
  • 不能保证每个process严格在同一个时刻(instant)记录状态，但是最终形成的需要是个**"有意义"状态**；
  • 快照需要被叠加(SuperImpose)到已有计算上，即尽量不干扰系统固有的计算。

3.2.2 算法的结束

  • 结束的含义：所有的 Process和Chan，都完成快照状态记录;
  • 如果是个强连通图，那么从任意一个节点开始，都能在有限时间内结束；
  • 如果不是强连通图，但是所有节点可达，那么从某些指定节点开始，也能在有限时间内结束。

3.3 结合token的例子讨论下算法

3.3.1 假设p要给q发送token，且p在q之前记录状态，随后发送了Marker给q

  1. 如果q收到的第一个Marker来自p，那么q的状态记录中包含了p进行记录之前的所有消息；

  2. 如果q收到的第一个Marker不是来自p，那么q等待收到p的Marker才算完成chan的状态记录。即生成的chan状态记录中包含了p在record之前的所有消息(在p record时，这些消息可能在chan中，也可能已经被q接收完)。

对于上面的例子，考虑p是发送token给q，p记录快照的时间点可以是：(A) In-p; (B). In-chan; (C). In-q。无论如何，记录的快照整体应该包含且只包含一个 token。

3.3.2 假设p给q发送token, 但是q先于p 记录状态

  • 如果q在收到token后才开始记录，即q记录时已经token传递完成，那么二者记录的都是token已经传递完成的一致状态;
  • 如果q在记录时，token在路上(in-chan)，那么q记录的状态，不含token，随后发送marker给p，但是此时q并没有完成chan的的快照记录；随后，p 记录自己的状态，并发送marker给 q，q才完成chan的状态记录。最终记录的是token出于in-chan状态，是个正确的状态记录。

与前面相比，为什么这里q有了in-chan? 因为 token位于q收到的两个Marker之间。而不是首个Maker之前。

总结

  • 无论p还是q先开始记录，最终都能得到一致的状态。
  • 在q收到首个Marker之前收到的token，计入了q自身的状态(已接收)；
  • 在 q收到首个Marker之后收到的token，计入了 chan 的状态。
